# TripTracer v0.0.1 Bootstrap Design

**Goal**

为 TripTracer 建立 `v0.0.1` 的项目初始化基线，明确 monorepo 目录职责、中间件镜像选型、端口规划与访问方式；当前轮次只沉淀文档，不下载镜像、不启动容器。

**Scope**

本次设计只覆盖文档层面的初始化约定：
- 镜像选型基线
- 端口与访问方式约定
- 中间件初始化参数约定
- 下一轮目录骨架与 `docker-compose.yml` 的落地范围

**Out of Scope**

- 不创建 `docker-compose.yml`
- 不创建实际目录或 `Dockerfile`
- 不实现前后端或算法代码
- 不执行任何容器拉取、构建、运行

**Architecture Direction**

仓库保持 monorepo 结构，应用层分为 `frontend`、`backend`、`algo` 三个独立子目录；基础设施层统一通过根目录 `docker-compose.yml` 编排。`v0.0.1` 先冻结运行基线，后续版本再按该基线逐步补齐目录、容器与应用代码。

**Chosen Images**

- Frontend build: `node:20-alpine`
- Frontend runtime: `nginx:1.27-alpine`
- Backend: `python:3.11-slim`
- Algo: `python:3.11-slim`
- MySQL: `mysql:8.0`
- Elasticsearch: `docker.elastic.co/elasticsearch/elasticsearch:8.13.4`
- Redis: `redis:7.2-alpine`
- MinIO: `minio/minio:RELEASE.2024-05-28T17-19-04Z`

**Access Baseline**

- Frontend: `http://127.0.0.1:8080`
- Backend: `http://127.0.0.1:8000`
- MySQL: `127.0.0.1:3306`
- Elasticsearch: `http://127.0.0.1:9200`
- Redis: `127.0.0.1:6379`
- MinIO API: `http://127.0.0.1:9000`
- MinIO Console: `http://127.0.0.1:9001`

**Decision Notes**

- 选用 `alpine`/`slim` 系列镜像以控制基础镜像体积，便于 Ubuntu 开发机和 Windows Docker 环境统一。
- Elasticsearch 先固定 8.x 单节点开发模式，减少集群复杂度。
- MinIO 独立暴露 API 与 Console 端口，便于后续对象存储联调与管理。

**Next Step**

下一轮按本文档约定创建目录骨架，并补齐 `docker-compose.yml` 中间件编排。
